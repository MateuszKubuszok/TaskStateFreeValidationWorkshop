Cats vs Scalaz

  • Cats - more focus on performance, modularity, less mature
  • Scalaz - more dogmatic approach, strong opinions among authors
  • in the beginning mostly political differences, now API very often differs

Concepts we'd better know

  • type classes
  • monads
  • implicits

3 flavors of polymorphism

  • subtyping: most often as interface inheritance
  • parametric polymorphism: generics, templates, parametrized classes/methods/functions
  • ad hoc polymorphism: function/operator overload

Type class

  • based on ad hoc polymorphism
  • puts constraint on a type, by requiring overloading of some functions
  • is not a class in OOP sense (!)

Metaphor of assembly line

  • having boxes of some sort
  • putting things into boxes
  • taking things out of the box by one, doing something with them and packing back into boxes of the same kind

Monad type class

  • type constructor
  • unit function (putting things into the box for the first time)
    def unit[T]: T => Monad[T]
  • bind function (taking something out of the box and deciding what to do with it)
    def bind[S, T]: (Monad[S], S => Monad[T]) => Monad[T]
    in Scala known as flatMap!
Mention the map -> flatten thing as well as where it doesn't apply (Future).

Monadic laws

  • left identity
    bind(unit(value), func) == func(value)
  • right identity
    bind(mValue, unit) == mValue
  • associativity
    bind(bind(mv, f1), f2) == bind(mv, v => bind(f1(v), f2))

Implicit parameters

  • can be used to extract implicit value from the scope, basing on its type
  • can be used to create a value from implicit function in scope basing, on its type
  • used (among others) to pass a type class to a method in both standard library (e.g. CanBuildFrom) as well as in Cats and Scalaz (where some type classes exists physically as objects)

Validation

Validation

Why

Existing ways of error handling

  • error flags/values
  • exceptions
  • Either and EitherT
  • exclusive or: Xor, \/

How - applicative functors

  • another type class
  • extends functor by handling functions stored within as a value
  • def pure[A]: A => Applicative[A]
  • def map[A, B]: (Applicative[A => B], Applicative[A]) => Applicative[B]

Syntax - wrapping values

Cats

val s: Validated[Error, Int] = valid[Error, Int](12)
val f: Validated[Error, Int] = invalid[Error, Int](error)

Scalaz

val s: Validation[Error, Int] = 12.success[Error]
val f: Validation[Error, Int] = error.failure[Int]

Syntax - applicative

Cats

def x: Validated[Error, Double]
def y = Validated[Error, Double => String]
def x: Validated[Error, String] = x.ap(y)

Scalaz

def x: Validation[Error, Double]
def y = Validation[Error, Double => String]
def x: Validation[Error, String] = x.ap(y)
Mention difference in signature: flipped arguments.

Syntax - cartesian builder

case class Test(i: Int, d: Double)

Cats

val result = (parseInt(x) |@| parseDouble(y)) map { Test }

Scalaz

val result1 = (parseInt(x) |@| parseDouble(y))(Test)
val result2 = parseDouble(y) <*> (parseInt(x) map { Test.curried })
Aka applicative builder.

Problem

How errors get aggregated?

Semigroups

  • algebraic structure as well as type class :)
  • basically value set with associative binary operator
  • def combine(a1: A, a2: A): A // alias a1 |+| a2
  • (a1 |+| a2) |+| a3 == a1 |+| (a2 |+| a3)
|@| - read like scream, "Admiral Ackbar" or "Macaulay Culkin".

Non empty list

Cats

final case class NonEmptyList[A](head: A, tail: List[A]) { ...
  def concat(other: NonEmptyList[A]): NonEmptyList[A] ... }

Scalaz

final class NonEmptyList[A](head: A, tail: IList[A]) { ...
  def append(f2: NonEmptyList[A]): NonEmptyList[A] ... }

Sequential validation

Cats

validated.andThen { value => anotherValudation(value) } // or
validated.ensure(errorMessage) { predicate }

Scalaz

import Validation.FlatMap._
validated.andThen { value => anotherValudation(value) } // or
validation.ensure(errorMessage) { predicate }
Explain that it will stop validation on first error.

Task

Task

Why

  • Scala's future is not a monad
  • Scalaz's future interface is impractical
  • useful functions for sync and async execution
  • trampoline let us avoid stack overflow
Mention that, while there are some important implementation concerns, but overall main advantage is the API.

For-comprehension

for {
  value  <- valueTask
  (a, b) <- pairTask(value)
  c      <- otherTask
} yield (a, b, c)
Mention that trampoline allows us to avoid stack overflow when we use flatMaps.

Exclusive or

Like Either but with bias towards right (monad!).

(value: \/[Left, Right]) match {
  case -\/(left)  => // used for errors
  case \/-(right) => // used for correct values
}

In Cats: Xor, Xor.Left and Xor.Right respectively.

Creating task

Task(operation)
Task.now(operation)
Task.delay(operation)
Explain that the latter will avoid using thread pool and that Task do NOT run immediately.

Running task

task.unsafePerformAsync {
  case -\/(error) => // handle left error
  case \/-(value) => // handle right value
}
task.unsafePerformSync // returns value or throws on error
task.unsafePerformSyncAttempt // returns Throwable \/ Value
Mention that timeout could be Duration or milliseconds in Long.

Running task with timeout

task.unsafePerformSyncFor(timeout)
task.unsafePerformSyncAttemptFor(timeout) // Throwable \/ Value

Running with interruptions

// returns () => Unit function for canceling task
task.unsafePerformAsyncInterruptibly {
  case -\/(error) => // handle left error
  case \/-(value) => // handle right value
}
// cancels task when atomicBoolean becomes true
task.unsafePerformAsyncInterruptibly(handler, atomicBoolean)

ExecutionServices and runner thread

val task1 = Task(function) // default ExecutionService
val task2 = Task(function)(es) // es service - might be implicit
val task3 = Task.fork(task1)(es) // forks ask1 on es service

Summary

  • nice interface
  • less stack-overflow
Mention monix's Task for people, who want higher performance.

State

State

Why - stateful calculations

var config = Config()
def x(): A = { config = update(config); calculateSth() }
def y(z: A): B = { config = update(config, z); calculateSth(z) }
val a = x
val b = y(a)

Why - less stateful

val config = Config()
def x(config: Config): (config, A) = {
  val config2 = update(config)
  config2 -> calculateSth()
}
def y(config: Config, z: A) = {
  val config2 = update(config, z)
  config2 -> calculateSth(z)
}
val (config2, a) = x(config)
val (config3, b) = y(config2, a)

Why - chained?

y.tupled(x(config)) // ??

How about?

for {
  a <- x
  b <- y(a)
} yield b

Creating State pipeline

def x = State[Config, A] { case (c, a) =>
  update(c) -> calculateSth() // (Config, A)
}
def y(z: A) = State[Config, B] { case (c, a) =>
  update(c, z) -> calculateSth(z) // (Config, B)
}
val pipe = for {
  a <- x
  b <- y(a)
} yield b // just a pipe! no effects yet!

Running State pipeline

Cats:

val (config2, b) = pipe.run(config).value // run once
val (config2, b) = pipe.run(config).memoize.value // memoize

Also runA and runS for just last value or state.

Scalaz:

val (config2, b) = pipe(config)

Also eval and exec for just last value or state.

Other useful methods

used for Cats Scalaz
s1 => (s2, a) apply apply
s => (s, a) pure state
s1 => (s2, ()) modify modify
s => (s, f(s)) inspect gets
s => (s, s) get get/init
_ => (s, ()) put put
This concerns State.x methods of the companion objects.

  • well create very simple State functions
  • again, we can look at tests and main to see how they perform

Summary

  • allows creating pipelines for stateful calculations
  • it can operate on immutable data, so we stay functional
  • calculations are only run when we supply our pipeline with initial state

Free

Free

Why

def getUser: User
def updateUser(user: User): User
def getUserPreferences(user: User): Set[Preference]
Explain problems with straightforward approach - composing, async, etc.

Why

def getUser: Future[User]
def updateUser(user: User): Future[User]
def getUserPreferences(user: User): Future[[Set[Preference]]
Explain limitations of using Future - stack overflow, running execution immediately.

Why

def getUser: Task[User]
def updateUser(user: User): Task[User]
def getUserPreferences(user: User): Task[[Set[Preference]]
Explain limitations of using Task - potentially stateful calculations, still another way of tight coupling with architecture.

Why

def getUser: State[Config, User]
def updateUser(user: User): State[Config, User]
def getUserPreferences(user: User): State[Config, [Set[Preference]]
Explain limitations of using Task - potentially stateful calculations, still another way of tight coupling with architecture.

How about

def getUser: X[User]
def updateUser(user: User): X[User]
def getUserPreferences(user: User): X[[Set[Preference]]

How about

val getUserWithPrefs = for {
  user  <- getUser
  prefs <- getUserPreferences(user)
} yield (user, prefs)

val x: Future[(User, Set[Preferences])] =
    getUserWithPrefs.interpret(asFuture)
val y: Task[(User, Set[Preferences])] =
    getUserWithPrefs.interpret(asTask)
val z: State[Config, (User, Set[Preferences])] =
    getUserWithPrefs.interpret(asState)
